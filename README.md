# Bitcoin і BTC: економічний та статистичний аналіз

[Читати онлайн-книгу](https://romankuznietsovbybuffmedic.github.io/bitcoin-economic-statistical-analysis/)

## Головне питання

> Чи може статистичний метод дати практично корисне правило торгівлі
> BTC після врахування витрат і ризику?

Книга написана для читача без спеціальної підготовки. Спочатку
пояснюється сенс, а формули й код додаються лише там, де вони потрібні.

## Що вже зроблено

- пояснено Bitcoin, BTC, BTC/USDT і спотову торгівлю;
- Binance перевіряється лише як початковий кандидат і має розриви;
- Bybit і Binance порівнюються на точно однаковому проміжку;
- Bybit обрано єдиним основним рядом;
- Bitstamp зберігається як незалежний контроль;
- однаково перевіряються пропуски, дублікати, OHLC і обсяг;
- підготовлено ціну, звичайну й логарифмічну дохідності;
- додано окремі інтерактивні графіки ціни й дохідностей;
- на обох графіках є повзунок для вибору часового проміжку;
- ціну показано для 0.001 BTC, або 100 000 сатоші, з одночасною
  підказкою про повну ціну 1 BTC;
- усі ряди на двох графіках незалежно вмикаються й вимикаються;
- увімкнені кнопки мають позначку `●` і синє тло, вимкнені мають
  позначку `○` і нейтральне тло;
- додано метадані джерел і SHA-256 маніфест локальних файлів;
- останній календарний рік зарезервовано для фінального тесту;
- дані хронологічно поділено на навчальну й тестову частини;
- перший рендер автоматично відновлює пакети й відсутні дані.

На цьому етапі не перевірено жодної статистичної гіпотези, не
побудовано моделі й не створено торгового правила. Прибуток, витрати та
ризик ще не оцінювалися. Це окремий наступний етап після формулювання
однієї вузької гіпотези.

## Де змінюються параметри

`config.yml` є єдиним місцем для:

- початку й цілої кількості календарних років у періоді;
- початкового кандидата для порівняння;
- основної та контрольної біржі;
- символів і валют;
- поля, яке визначає ціну;
- базового інтервалу;
- тривалості фінального тесту;
- горизонту прогнозу й частоти переоцінювання моделі;
- частки BTC, для якої показується вартість на графіку ціни;
- кількості паралельних процесів;
- дозволу на автоматичне завантаження даних перед рендером.

Кінець періоду обчислюється як
`study.data_start + study.data_years`. Скрипти та розділи книги читають
ці значення, а не повторюють їх.
Поточний конвеєр отримання підтримує базові свічки `1h`. Більші
інтервали потрібно буде будувати агрегуванням годинного ряду.

## Архітектура

- `R/` містить функції без запуску всього дослідження;
- `scripts/` отримує, перевіряє та готує дані;
- `*.qmd` читають готові дані й будують книгу;
- `data/` створює локальні кеші, які не зберігаються в Git;
- `data-manifest.yml` локально фіксує SHA-256 поточних наборів;
- `renv.lock` фіксує версії R-пакетів.

## Перший запуск із GitHub

Потрібні встановлені R 4.6.1, Quarto та доступ до інтернету.
У Linux також можуть знадобитися стандартні засоби компіляції
R-пакетів. Системні програми й бібліотеки проєкт не встановлює.

Архів GitHub навмисно не містить RDS, CSV або інших ринкових даних.
Після завантаження й розпакування відкрийте термінал у корені проєкту
та виконайте одну команду:

```bash
quarto render
```

Перед побудовою книги Quarto автоматично запускає
`scripts/bootstrap_project.R`. Він:

1. встановлює зафіксовану версію `renv`, якщо її ще немає;
2. відновлює точні версії пакетів із `renv.lock`;
3. завантажує Binance для порівняння, Bybit для основного аналізу та
   Bitstamp для контролю;
4. готує ціну, дохідності та часовий поділ;
5. перевіряє метадані, OHLC, часові межі, SHA-256 і річний тест;
6. лише після успішних перевірок будує книгу в `_book/`.

Перший запуск може бути тривалим через установлення пакетів і
завантаження багаторічних погодинних даних. Якщо він перерветься,
повторіть `quarto render`: уже перевірені локальні файли буде
використано повторно.

Під час завантаження консоль показує поточну біржу, відсоток,
кількість опрацьованих частин даних і стани `[ГОТОВО]` або
`[КЕШ ГОТОВИЙ]`. Це реальний прогрес мережевих запитів, а не
анімація.

Наступні рендери не звертаються до бірж, якщо локальні файли
відповідають `config.yml` і проходять перевірку.

## Ручне керування

Для примусового повторного отримання всіх даних:

```bash
Rscript scripts/refresh_all_data.R
```

Для окремої перевірки джерел, SHA-256 і часового поділу:

```bash
Rscript scripts/check_data_and_split.R
```

Зміна лише `evaluation.test_years` не потребує повторного
завантаження. Після зміни основної біржі, початку або кількості років
підготовлений набір перебудується автоматично. Зміна джерела, символу,
базового інтервалу або часових меж автоматично позначить несумісні
кеші для нового завантаження.

Щоб повністю заборонити мережеві запити під час рендеру, задайте
`runtime.auto_download_data: false` у `config.yml`. Якщо потрібних
даних немає, рендер тоді зупиниться з прямим повідомленням.
