# Bitcoin і BTC: економічний та статистичний аналіз

[Читати онлайн-версію дослідження](https://romankuznietsovbybuffmedic.github.io/bitcoin-economic-statistical-analysis/)

Відтворюване дослідження ринку BTC за допомогою R, Quarto та Git.

## Мета проєкту

Проєкт перевіряє, чи може статистичне моделювання BTC дати практично корисне рішення для spot-торгівлі після врахування витрат і ризику.

Головне питання:

> Чи додає статистичне моделювання BTC практичну користь для spot-рішення після витрат і на даних, яких модель не бачила під час побудови?

Мета не полягає у використанні всіх доступних моделей. Кожна модель допускається до роботи лише тоді, коли зрозуміло, яку властивість вона описує, яке рішення може змінити та як перевірятиметься її додаткова користь.

## Практичні межі

- основний ринок: Binance Spot `BTCUSDT`;
- основний часовий інтервал: `1d`;
- додатковий інтервал: `4h`;
- контрольний інтервал: `1h`;
- тільки spot без кредитного плеча;
- рішення про частку капіталу в BTC або USDT;
- обов'язкове врахування комісії, спреду, прослизання та обороту;
- перевірка на майбутньому часовому відрізку, якого модель не бачила.

За малого початкового капіталу проєкт не може створити відчутний стабільний дохід. Його практична мета - перевірити наявність торгової переваги, навчитися контролювати ризик і створити якісне статистичне дослідження.

## Мінімальне практичне дослідження

Перший завершений експеримент включатиме:

1. базові рішення: постійне утримання BTC, відсутність позиції та просте правило історичної волатильності;
2. першу модель: звичайний GARCH для прогнозу умовної волатильності;
3. правило зміни частки BTC залежно від оціненого ризику;
4. послідовне тестування на нових даних;
5. урахування торгових витрат;
6. оцінювання чистої дохідності, максимальної просадки, обороту та стабільності;
7. рішення про доцільність складнішої моделі лише після результатів простої.

Перша перевірювана гіпотеза полягає в тому, що прогноз волатильності може допомогти змінювати частку BTC і контролювати просадку, навіть якщо напрямок наступної зміни ціни не прогнозується.

## Що планується далі

1. описати дохідність, волатильність, хвости розподілу та просадки;
2. перевірити стаціонарність, автокореляцію й кластеризацію волатильності;
3. побудувати базове правило історичної волатильності;
4. оцінити GARCH і перевірити його практичну додаткову користь;
5. виконати послідовну перевірку на нових даних;
6. застосувати витрати відповідно до обороту стратегії;
7. вирішити, чи є підстава переходити до MS-GARCH або іншої складнішої моделі.

## Правило вибору моделей

Модель використовується лише тоді, коли:

1. сформульовано конкретне питання;
2. зрозуміло, що саме оцінює модель;
3. проста базова модель є недостатньою;
4. результат можна перевірити на нових даних;
5. покращення залишається після витрат;
6. результат має практичне, а не лише статистичне значення.

Відсутність прогнозованості або нездатність складної моделі перевершити просте правило також є змістовним результатом.

- `references.bib` і `references.qmd` - бібліографія.

## Відтворення

Потрібні R, Quarto, Git і пакет `renv`.

Відновлення середовища:

```r
renv::restore()
```

Рекомендований порядок першого запуску:

```r
source(
  "scripts/03_update_binance_fast.R",
  encoding = "UTF-8"
)
```

Цей сценарій завантажує та перевіряє Binance, створює компактні `1h`, `4h`, `1d` і не тримає весь хвилинний ряд у пам'яті.

Побудова денного набору для моделювання:

```r
source(
  "scripts/05_build_model_data.R",
  encoding = "UTF-8"
)
```

Результат зберігається у `data/model/binance/spot/BTCUSDT/` як RDS, CSV, словник колонок, метадані, опис часового поділу та звіт якості.

Побудова книги:

```bash
quarto render
```

HTML-книга створюється в каталозі `_book/`.
