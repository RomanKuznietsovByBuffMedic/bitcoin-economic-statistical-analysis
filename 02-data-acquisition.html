ImportantГоловне питання

Який погодинний ряд BTC достатньо повний, щоб використовувати його для графіків, статистичного аналізу, моделі та торгового правила без прихованого заповнення пропусків?

2.1 Binance як початкове джерело

Спочатку для проєкту було обрано Binance Spot BTC/USDT. Це спотовий ринок, де ціна показує, скільки USDT коштує один BTC. Дані отримуються з офіційних місячних архівів Binance з перевіркою SHA-256, а неповний останній місяць і знайдені розриви повторно перевіряються через REST API [1], [2].

Один рядок даних відповідає одній завершеній годині UTC. Для кожної свічки використовуються:

  • open - ціна першої угоди;
  • high - найбільша ціна;
  • low - найменша ціна;
  • close - ціна останньої угоди;
  • volume - обсяг BTC.

Ціна \(P_t\) у наступному розділі визначається як close. Незавершена поточна година до набору не включається.

2.1.1 Період і формат дати

Початок задається у config.yml, а кінець автоматично обчислюється як початок плюс ціла кількість календарних років.

NoteЯк читати дату і час

Формат YYYY-MM-DD HH:MM:SS означає:

  • YYYY - рік;
  • MM - місяць;
  • DD - день;
  • HH - година;
  • MM після пробілу - хвилини;
  • SS - секунди.

Наприклад, 2021-07-06 00:00:00 означає 6 липня 2021 року, початок доби за UTC.

TipЧи поширюється зміна config.yml на всю роботу?

Так. Розділи книги та сценарії даних викликають read_project_config().

Якщо змінити study.data_start, study.data_years, біржу, символ, інтервал або інше джерельне значення, старі кеші перестануть відповідати метаданим. Перед рендером вони будуть отримані заново, а підготовлений набір, таблиці, часовий поділ і графіки перебудуються.

Кінцеву дату не потрібно змінювати окремо: вона завжди обчислюється як study.data_start + study.data_years. Параметри, яких немає в config.yml, наприклад текст пояснень або формули, автоматично не змінюються.

2.1.2 Перевірка повноти Binance

Для кожної біржі одна функція перевіряє:

  • потрібні поля та часові позначки UTC;
  • точний початок і кінець періоду;
  • дублікати та пропущені години;
  • додатні числові OHLC-ціни;
  • співвідношення low ≤ open, close ≤ high;
  • невід'ємний обсяг.

Для Binance внутрішні розриви не приховуються і не зупиняють побудову цього розділу. Вони є результатом аудиту, який потрібно показати.

2.2 Bybit як альтернатива на тому самому проміжку

Наступним кандидатом є Bybit Spot BTC/USDT. Порівняння чесне, тому що Binance і Bybit мають:

  • той самий спотовий ринок;
  • ту саму пару BTC/USDT;
  • той самий погодинний інтервал;
  • ті самі початкову й кінцеву межі;
  • ту саму очікувану кількість годин.

Bybit отримується через офіційний Kline API [3].

2.3 Вибір основного джерела

ImportantРішення після однакового аудиту

Основним рядом стає Bybit Spot BTC/USDT.

Binance зберігається в config.yml, локальному кеші, маніфесті та цьому розділі лише для відтворюваного початкового порівняння з Bybit. Воно не використовується для підготовленого набору, графіків ціни й дохідностей, навчання моделі або торгового правила.

Підготовлений файл будується лише з кешу біржі, визначеної в study.primary_exchange. Зараз це Bybit. Пропуски Binance не заповнюються даними Bybit, а дані двох бірж не змішуються.

Сам вибір повного джерела ще не доводить можливість заробітку. Він лише усуває відому технічну проблему перед статистичним аналізом.

2.4 Навіщо потрібна контрольна біржа

Після вибору Bybit потрібна ще одна повна й незалежна біржа. Її головна роль з'явиться після побудови моделі та торгового правила.

На Bybit виконуватимуться:

  • побудова ознак;
  • вибір моделі;
  • налаштування параметрів;
  • коригування порогів торгового правила;
  • аналіз помилок у навчальній частині.

Після цього модель і правило фіксуються. На контрольну біржу без змін переносяться:

  • спосіб обчислення ознак;
  • навчена на Bybit модель без перенавчання;
  • пороги відкриття та закриття позиції;
  • напрям сигналу;
  • правило розміру позиції;
  • порядок обчислення результату.
WarningКонтроль не можна перетворювати на нове навчання

Критерії прийнятної різниці результатів потрібно визначити до перегляду фінального результату контрольної біржі.

Якщо після першої перевірки змінити модель або правило з урахуванням контрольного результату, ця біржа більше не буде незалежним контролем.

Якщо незмінені модель і правило дають співрозмірний результат на іншій біржі, це буде аргументом, що вони вловили закономірність руху BTC, а не лише особливість Bybit. Це не абсолютний доказ, але сильніша перевірка, ніж тест лише на одній біржі.

Контрольна біржа має і додаткову роль ще до моделювання. Вона допомагає виявити зміщення часу, переплутані поля або іншу помилку, яка могла створити повний, але неправильний ряд Bybit.

2.5 Bitstamp як контрольна біржа

Контрольним рядом є Bitstamp Spot BTC/USD, отриманий через офіційний OHLC API [4].

Bitstamp і Bybit перевіряються на тому самому 5-річному погодинному проміжку. Для обох джерел повнота є обов'язковою: якщо хоча б одна година відсутня, рендер зупиняється.

2.5.1 Поточна технічна перевірка даних

Bybit показує BTC/USDT, а Bitstamp показує BTC/USD. Тому рівні цін не повинні точно збігатися. Для поточного технічного аудиту порівнюються лише однакові години та погодинні логарифмічні дохідності:

\[ r_t=\ln\left(\frac{P_t}{P_{t-1}}\right). \]

Показати / приховати код 
agreement <- compare_hourly_exchanges(

  reference = reference_quality$data,
  primary_name = config$primary$name,
  reference_name = config$reference$name
)
agreement_values <- agreement$summary$Значення
agreement_table <- tibble::tibble(
  `Показник` = agreement$summary$Перевірка,
  `Результат` = c(
    format_count(agreement_values[[1]]),
    format_count(agreement_values[[2]]),
    paste0(format_decimal(agreement_values[[3]], 6), " %"),
    paste0(format_decimal(agreement_values[[4]], 6), " %"),
    paste0(format_decimal(agreement_values[[5]], 6), " %"),
    format_decimal(agreement_values[[6]], 6)
  )
)

2.6 Як джерела та кеші контролюються

Перед кожним рендером scripts/ensure_project_data.R звіряє локальні RDS із config.yml. Мережевий запит виконується лише тоді, коли потрібного файла немає, він не читається, його походження не підтверджено або метадані не відповідають конфігурації.

У таблиці показано короткі назви посилань замість довгих URL.

2.7 Які дані використовуються далі

ImportantТри джерела мають три різні ролі
  • Binance Spot BTC/USDT використовується лише в цьому розділі для відтворюваного порівняння повноти з Bybit.
  • Bybit Spot BTC/USDT є єдиним джерелом підготовлених даних, графіків, розробки моделі й коригування торгового правила.
  • Bitstamp Spot BTC/USD зберігається окремо для технічного аудиту та майбутньої одноразової перевірки незмінених моделі й правила.

Останній календарний рік Bybit уже зарезервовано для фінального часового тесту. Bitstamp не використовується для вибору моделі, параметрів або торгового правила. Комісію, спред, ліквідність і прослизання буде враховано на етапі практичного тесту [5], [6].

2.8 Висновок

Спочатку перевірено Binance. Графік і таблиця показують його розриви на тому самому повному багаторічному проміжку, на якому перевірено Bybit. Оскільки Bybit має повний ряд, воно обране єдиним основним джерелом.

Після цього Bitstamp введено як незалежний контроль. Зараз він додатково перевіряє часову й числову узгодженість даних. Пізніше на ньому один раз застосовуватимуть ту саму навчену на Bybit модель і те саме торгове правило без коригування. Дані трьох бірж не змішуються.
